# SKY-KING — זיהוי כתב יד מודע-הקשר (Offline) — המלצות ארכיטקט

> מסמך **המלצות לפני יישום**. מבוסס מחקר online (כל עוד יש גישה) + סקירת הקוד הקיים.
> אילוץ-על: **המערכת רצה ברשת סגורה ללא אינטרנט** → כל פתרון חייב להיות 100% offline (בלי CDN/ענן/הורדות בזמן ריצה). עודכן: 2026-06-23.

---

## 1. המטרה
פקח/בקר כותב בעט על מסך מגע; הכתב **מזוהה והופך לטקסט לפי ההקשר**. דוגמת-על: כותבים או"ק על המפה → אם קיים פ"מ תואם ברשימה, הוא "נגרר" למפה אוטומטית. בונים מנגנון **כללי**, ואז מאפיינים כל מקרה.

## 2. מה קיים היום (ומה הבעיה)
| רכיב | מצב | בעיה לרשת סגורה |
|------|-----|------------------|
| `HandwritingOverlay.tsx` | קנבס + ציור + `Tesseract.createWorker('eng')` | 🔴 **שבור offline** — Tesseract מוריד worker+wasm+traineddata מ-CDN |
| `tessedit_char_whitelist:'0123456789'` | מזהה **ספרות בלבד** דרך OCR של תמונה | מוגבל; OCR של כתב יד חלש (במיוחד עברית) |
| `learned_digits` (DB) + `utils/digits.ts` | שומר **תמונת** כל ספרה שהפקח צייר (per crew) | רעיון הפרסונליזציה טוב — אבל מבוסס תמונה, לא מבוסס strokes |
| לכידת strokes ב-pointer events | קיימת — אך **נזרקת** (מרונדרת לתמונה ל-Tesseract) | מבזבזים את המידע הכי טוב (מסלול העט) |

> מסקנה: היום מזהים **תמונה** (OCR offline-handwriting) ותלויים ב-CDN. הכיוון הנכון הפוך.

## 3. תובנה ארכיטקטונית מרכזית
1. **יש לנו עט → יש לנו strokes** (מסלול נקודות x,y,t). זיהוי מבוסס-strokes ("online handwriting") **מדויק ומהיר בהרבה** מ-OCR של תמונה, ובעיקר — קל למימוש offline ו**אגנוסטי לשפה** (עברית/לטינית/ספרות).
2. **ההקשר מצמצם את מרחב התשובות.** לרוב לא צריך זיהוי תווים מושלם — צריך להתאים לקבוצה **ידועה וקטנה**: או"קים ברשימת הפ"מ, שמות בסיסים, ספרות. זיהוי גס + **התאמה מטושטשת (fuzzy)** מול הקבוצה = דיוק גבוה מאוד.

## 4. אפשרויות זיהוי offline שנבדקו

| גישה | offline נקי? | עברית? | משקל | למידה מדגימות | התאמה |
|------|--------------|--------|------|----------------|--------|
| **$P / $N / $1 Point-Cloud Recognizer** (UW/Wobbrock) | ✅ JS טהור, בלי מודל/CDN | ✅ אגנוסטי לשפה (מתאים צורות) | זעיר (~קובץ אחד) | ✅ 5+ דגימות → >99% user-dependent | ⭐ **מומלץ** |
| **TensorFlow.js / ONNX Runtime Web** + מודל bundled | ✅ אם המודל ארוז מקומית | תלוי במודל (צריך מודל עברית) | כבד (MB) + צריך לאמן/להשיג מודל | מודל מאומן מראש | חלופה/עתיד |
| **Tesseract.js** (wasm+traineddata ארוזים מקומית) | ✅ רק אם ארוז (לא ברירת מחדל!) | חלש לכתב יד עברי | בינוני-כבד | ❌ | fallback לטקסט מודפס |

> **מקורות:** $P/$1/$N Recognizers (depts.washington.edu/acelab/proj/dollar — קוד JS), TensorFlow.js handwriting, ONNX Runtime Web.

## 5. הארכיטקטורה המומלצת — 3 שכבות (DRY, מודולרי)

```
┌─ שכבת לכידת דיו (Ink Capture) ─ רכיב משותף, הרחבת HandwritingOverlay
│   Pointer Events → strokes [{x,y,t,pressure,pointerType}]  (לשמור! לא לרנדר לתמונה)
│   זיהוי עט מול אצבע (pointerType==='pen'); palm rejection בסיסי
├─ שכבת זיהוי (Recognition) — offline, pluggable
│   ראשי:  $P/$N על strokes + templates per-crew (מחליף את learned_digits בגרסת strokes)
│   טקסט רב-תווי: סגמנטציה לתווים (לפי רווח/זמן) → זיהוי כל תו → הרכבת מחרוזת
│   fallback: Tesseract.js ארוז מקומית (אם בכלל) / CNN זעיר לספרות (tfjs bundled)
└─ שכבת פתרון-הקשר (Context Resolver) — הלב החכם
    קלט: candidates מהזיהוי + ההקשר הפעיל
    התאמה מטושטשת (Levenshtein/Jaro-Winkler) מול הקבוצה הידועה של ההקשר
    → פעולה (למשל: הבא פ"מ למפה כאילו נגרר). סף ביטחון + UI הבהרה כשיש כמה מועמדים.
```

### פרסונליזציה ולמידה — **משולב (בסיס + למידה מתגלגלת)** ✅ הוחלט
> אין enrollment כפוי. המערכת עובדת **מיד** לכל משתמש, ולומדת אותו **אוטומטית תוך כדי שימוש**.

- **ערכת בסיס (seed):** templates התחלתיים לספרות + א"ב עברי + לטיני, מסופקים עם המערכת → עובד from day one בלי שאף אחד יכתוב מראש.
- **למידה מתגלגלת (passive):** כל פעם שזיהוי **מאושר/מתוקן** ע"י המשתמש, ה-strokes נשמרים כ-template אישי (per crew_member). הדיוק עולה עם השימוש, בלי סשן אימון ייעודי.
- **אחסון:** טבלה `learned_strokes` (או הרחבת `learned_digits`): `{ token, strokes JSONB, crew_member_id, source: 'seed'|'user' }`. הכל ב-Postgres המקומי, offline.
- **דירוג זיהוי:** templates אישיים מקבלים משקל גבוה יותר מ-templates הבסיס → התאמה אישית בלי לאבד את הכיסוי הגנרי.
- **כיול מהיר ע"י פסקה (מומלץ, אופציונלי):** המערכת מציגה למשתמש **פסקת כיול קצרה אחת** שמכסה את *כל* התווים — פנגרם עברי (כל 22 האותיות) + 5 סופיות (ך ם ן ף ץ) + ספרות 0–9 + הסימנים/אופרטורים שבשימוש (°, /, -, ', "). המשתמש כותב אותה ברצף טבעי.
  - יתרון מרכזי: **הטקסט הצפוי ידוע מראש** → כל קבוצת strokes מתויגת אוטומטית לתו הנכון, בלי תיקון ידני ובלי "כתוב כל אות X פעמים". סשן יחיד קצר = כיסוי מלא של כתב היד האישי.
  - דוגמת פנגרם עברי לכיול: "דג סקרן שט בים מאוכזב ולפתע מצא חברה" (לאמת כיסוי מלא; להוסיף סופיות/ספרות/סימנים בשורה נפרדת).
  - נשאר **אופציונלי**: בלי כיול עובדים על ערכת הבסיס + למידה מתגלגלת; עם כיול — דיוק אישי גבוה כבר מההתחלה.

### למה זה עונה בדיוק על האילוצים
- **100% offline:** $-recognizer הוא JS טהור — בלי מודל, בלי CDN, בלי הורדות. ה-templates ב-DB שלנו.
- **אופטימלי לעט:** משתמש ב-strokes (לא תמונה) → מדויק יותר offline.
- **עברית בחינם:** ה-recognizer מתאים צורות (point-clouds), אגנוסטי לשפה — אין צורך במודל OCR עברי.
- **הקשר = דיוק:** קבוצות מועמדים קטנות וידועות + fuzzy match.
- **DRY:** מרחיב את `HandwritingOverlay` ואת רעיון הלמידה הקיים, לא משכפל.

## 6. תוכנית מדורגת (לאחר אישור)
1. **Spike/POC** (feature branch): הוספת $P (קובץ JS bundled) + לכידת strokes ב-overlay; זיהוי ספרות מול templates שנלמדו offline — להוכיח דיוק בלי CDN. **בדיקת VERIFY:** עובד עם DevTools→Offline.
2. **שכבת Context Resolver גנרית** + API קטן לקבלת קבוצת המועמדים לפי הקשר (callsigns/bases). fuzzy match + UI הבהרה. TDD ללוגיקת ההתאמה.
3. **המקרה הראשון (אפיון):** "או"ק על המפה → גרירת פ"מ" — מימוש מלא end-to-end.
4. **Enrollment UI + `learned_strokes`** (מיגרציה דרך `/migrate`) + seed templates.
5. הרחבה למקרים נוספים (שדות מספריים, שמות בסיסים...) — כל אחד = הגדרת candidate-set + פעולה.

## 7. סיכונים והקלות
- **Tesseract offline:** ברירת המחדל מורידה מ-CDN → אם נשאר כ-fallback, **חובה לארוז** wasm+traineddata מקומית ולהצביע ל-paths מקומיים. עדיף לעבור ל-strokes ולא להישען עליו.
- **כתב עברי רציף/מחובר:** סגמנטציה קשה → הקלה: UX של "טוקן אחד בכל פעם" / תאי-כתיבה (cell per char) + הסתמכות על התאמת-הקשר; ולוקבולרי ידוע → templates של טוקנים שלמים.
- **עלות enrollment:** צריך דגימות ראשוניות per-user → הקלה: ערכת בסיס + למידה מצטברת מתיקונים.
- **זיהוי עט מול כף יד:** `pointerType` + palm rejection.

## 8. שורה תחתונה
מומלץ: **$P/$N (strokes, offline) + שכבת Context-Resolver עם fuzzy-match מול קבוצות ידועות**, על תשתית ה-overlay והלמידה הקיימים. זה הפתרון הקליל, המדויק והמתאים ביותר לרשת סגורה — ומנצל שיש עט.

> סטטוס: **Phase 1 (ליבה גנרית) יושם ומוזג** — `dollarRecognizer.ts`, `handwritingContext.ts`, `HandwritingPad.tsx`, `handwritingTemplates.ts` (21 בדיקות).

## 9. אפיון מקרה #1 — "או"ק על מפת הבקר → גרירת פ"מ"

### סטוריית משתמש
כבקר, אני **כותב בעט או"ק כלשהו ישירות על מפת הבקר**; המערכת מזהה אותו, מאתרת את הפ"מ התואם ברשימה, ו**ממקמת אותו על המפה במקום שבו כתבתי** — כאילו גררתי אותו לשם. כך אני מביא פ"מ למפה בלי לחפש ולגרור.

### נקודות חיבור בקוד (אומת)
| מה | היכן | שימוש |
|----|------|-------|
| מיקום פ"מ על המפה | `handleMove(id, x, y, toMap, pinX, pinY, zone…)` ב-SectorDashboard | **לקרוא לה כמו שהיא** — מחשבת geo מ-pin (`imagePctToGeo`), מעדכנת `map_pin_x/y`+`map_lat/lon`, שומרת `PUT /api/strips/:id`. אפס לוגיקת מיקום חדשה (DRY) |
| עיגון המפה | `mapGeoAnchorRef`, `mapImgBoundsRef`, `currentMapId` | כבר קיימים — ה-pad יקבל אותם |
| קבוצת המועמדים | `strips[].callSign/callsign` | הקבוצה ל-Context-Resolver = או"קים של פ"מים שעוד **לא על המפה** |
| רינדור הסיכה | קוד קיים ב-~9034 (`geoToImagePct`) | אוטומטי אחרי `handleMove` |

### החלטת UX: **מצב עט אחד** (לא toggle חדש) ✅
כבר קיים מצב ציור עם סרגל כלים (`drawTool: 'pen'|'eraser'|'circle'|'rect'`, כפתור העט בצד שמאל של המפה) שכבר **לוכד strokes** (`currentStrokeRef`, `isDrawingRef`, handlers ב-~4588). לכן **לא מוסיפים מצב עט נוסף** — מוסיפים **כלי חמישי לסרגל הקיים**: `'recognize'` (אייקון ✍️/או"ק). כשהוא פעיל, ה-stroke לא מצויר כאנוטציה קבועה אלא מוזן למנוע הזיהוי. שימוש חוזר מלא בתשתית העט הקיימת.

### הזרימה (end-to-end)
1. הבקר בוחר את כלי **"זיהוי" (✍️)** בסרגל הציור הקיים (אותו מצב עט).
2. כותב או"ק **במקום כלשהו על המפה**. נלכדים strokes (דרך `currentStrokeRef` הקיים) + מיקום (centroid הדיו ב-image-px). תמיכה ברב-stroke: צבירת strokes עד **idle קצר (~800ms)** מאז ה-stroke האחרון, או כפתור "✓ זהה".
3. `$P` מזהה → מחרוזת גסה. `resolveContext` מתאים מטושטשת מול או"קים שלא-על-המפה.
4. **תוצאה:**
   - התאמה ברורה (best, לא ambiguous, מעל סף) → ממקם מיד: `handleMove(stripId, x, y, true, pinX=centroid.x, pinY=centroid.y)`. הבזק/טוסט "נוסף: <או"ק>" + **Undo** קצר.
   - ambiguous (כמה קרובים) → פופ-אפ הבהרה: רשימת מועמדים מדורגת (tap לבחירה).
   - אין התאמה מעל סף → טוסט "לא זוהה — נסה שוב" (לא יוצר פ"מ; יצירה היא מקרה נפרד).
5. בעת אישור → ה-strokes נשמרים כדגימת **למידה מתגלגלת** (per crew) לשיפור עתידי.

### UX (לפי `/atc-ux`)
- **מהירות/זמן מבט:** פעולה אחת (כתיבה) מחליפה חיפוש+גרירה. משוב מיידי (≤100ms ויזואלי על ה-stroke).
- **אישור לפעולה:** מיקום אינו הרסני → לא צריך modal; מספיק טוסט + Undo (תואם עקרון "לא modal מיותר").
- **שקיפות:** הטוסט אומר מה זוהה ולאן; Event Log רושם את ההוספה.
- **עט מול אצבע:** `pointerType==='pen'` להפעלת לכידה; אצבע = pan רגיל (palm rejection).
- **הבהרה:** כשיש ambiguity — להציג מועמדים גדולים וברורים לבחירה מהירה.

### נתונים
- **אין שדות חדשים בפ"מ** — נעשה שימוש ב-`map_pin_x/y`, `map_lat/lon` הקיימים דרך `handleMove`.
- **חדש:** טבלת `learned_strokes` (דרך `/migrate`) ללמידה — `{label, strokes JSONB, crew_member_id, source}`. ב-Phase זה אפשר להתחיל בלמידה בזיכרון/localStorage ולחבר DB מיד אחרי.

### מקרי קצה
- פ"מ כבר על המפה → "העבר" למיקום החדש (אותה `handleMove`).
- כמה פ"מים עם או"ק זהה → להציג את כולם להבהרה.
- כתב לא קריא / מתחת לסף → טוסט, ללא שינוי.
- מפה ללא anchor → אי-אפשר geo; למקם לפי pin בלבד (כמו גרירה רגילה ללא anchor).

### Acceptance Criteria
- [ ] כתיבת או"ק קיים על המפה → הפ"מ מופיע על המפה במיקום הכתיבה, עם geo נכון (`map_lat/lon`).
- [ ] או"ק לא קיים/לא קריא → אין שינוי + משוב ברור.
- [ ] ambiguity → פופ-אפ בחירה; בחירה ממקמת.
- [ ] Undo מחזיר את הפ"מ מהמפה.
- [ ] עובד **offline** מלא (DevTools→Offline).
- [ ] אצבע ממשיכה ל-pan; רק עט כותב.
- [ ] אישור → נשמרת דגימת למידה.

### תכנית יישום (Phase 2)
1. **הוספת כלי `'recognize'`** ל-enum `drawTool` ולסרגל הציור הקיים (אייקון ✍️) — שינוי מינימלי, בלי מצב עט חדש.
2. `useMapHandwriting` hook (מבודד) — מקבל strokes שנצברו + `anchor/bounds/candidates`, מריץ `$P`+`resolveContext`, מחזיר `onResolved`. הלוגיקה **מחוץ** למונוליט.
3. בענף ה-stroke-end הקיים: אם `drawTool==='recognize'` → לצבור strokes (idle ~800ms/כפתור ✓) → להעביר ל-hook במקום לצייר.
4. חיבור ל-`handleMove(stripId, x, y, true, centroid.x, centroid.y)` + טוסט + Undo + פופ-אפ הבהרה.
5. `learned_strokes` (migrate) + שמירת דגימה באישור.
6. VERIFY: offline mode, ובדיקת זרימה מול נתונים אמיתיים; `/qa` + `/atc-ux`.

> נוגע ב-SectorDashboard (קובץ ענק) → הלוגיקה ב-`useMapHandwriting` hook; ב-SectorDashboard רק הוספת ערך ל-enum + ענף קצר ב-handler הקיים. לא להזריק לוגיקה לתוך המונוליט.
